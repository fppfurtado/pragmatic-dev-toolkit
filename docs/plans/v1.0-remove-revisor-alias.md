# v1.0 — remover alias `{revisor: ...}`

## Contexto

Em v0.11.0 o plugin formalizou `{reviewer: ...}` (EN) como mecânica canônica das anotações de revisor em planos e marcou `{revisor: ...}` (PT) como alias deprecado, com promessa de remoção em v1.0 (CHANGELOG entries 21, 39–40 da v0.11.0; `docs/philosophy.md:139` antes deste cutover). Em v0.11.1 o alias seguia prose-driven-aceito por `/run-plan` com warning amigável.

Este plano executa o cutover. Decisão de timing: **cortar direto v0.11.x → v1.0** (sem shipar v0.12 sentinel). A janela contratada `v0.11–v0.12` é reduzida para `v0.11.x` apenas, justificada no `### Notes` do CHANGELOG da v1.0 como mudança consciente de contrato — v0.12 sem mudanças reais seria overhead sem ganho dado o uso externo presumivelmente baixo.

Inconsistência colateral aproveitada para limpeza: `docs/philosophy.md:33` ainda usava `{revisor: qa}` / `{revisor: security}` como exemplo na tabela path contract — escapou da migração da v0.11.0. Vai junto.

Não há código Python a alterar — o tratamento da anotação é 100% prose-driven (skill markdown interpretada pelo LLM no runtime). Cutover é puramente editorial + bump de versão.

## Resumo da mudança

Bump major (0.11.1 → 1.0.0) por quebra de contrato em planos externos que ainda usem `{revisor:}`. Quatro arquivos editados, dois manifests bumpados; este próprio plano arquivado em `docs/plans/` como registro do cutover.

Planos arquivados em `docs/plans/` (`v0.11-coerencia-desambiguacao.md`, `desacoplar-path-contract.md`, `gap-bifurcacao-arquitetural.md`, `gen-tests-python.md`, `qa-and-security-reviewers.md`) **não mudam** — são registros históricos de releases já shipadas. Idem CHANGELOG entries de releases passadas.

## Notas operacionais

- **Sequência de commits** (matching pattern observado em git log: separar conteúdo de bump):
  1. `chore(skills): drop {revisor:} alias acceptance from /run-plan` — edita `skills/run-plan/SKILL.md`.
  2. `docs(philosophy): drop deprecated {revisor:} alias paragraph; fix path-contract example` — edita `docs/philosophy.md`.
  3. `docs(install): add smoke test for {revisor:} rejection` — edita `docs/install.md`.
  4. `docs: record the v1.0 plan in docs/plans/` — adiciona este arquivo.
  5. `chore(release): bump version to 1.0.0` — edita `plugin.json`, `marketplace.json`, `CHANGELOG.md`.

- **Tag e push**: após o último commit, `git tag v1.0.0 && git push origin main && git push origin v1.0.0` — mesmo procedimento das releases anteriores (v0.11.1 mais recente).

- **Risco baixíssimo**: o único cenário de regressão é operador externo com plano contendo `{revisor:}` invocando `/run-plan` pós-v1.0. Mitigação: a recusa é explícita e textual (não silenciosa), redirecionando para `{reviewer:}`. Custo de migração externa é trivial (find/replace).

- **Plano in-flight, não post-hoc**: o pattern típico do repo (`docs: record the vX.Y.Z plan in docs/plans/` como commit isolado depois do bump) sugere registro post-hoc. Este plano é exceção — criado durante a execução para servir como registro do cutover, ainda assim na sequência correta (antes do bump final).
