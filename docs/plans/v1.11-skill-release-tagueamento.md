# v1.11 — skill `/release` para tagueamento

## Contexto

O plugin já tem disciplina explícita em torno de **commits** ao longo do fluxo: `/run-plan` produz micro-commits (um por bloco do plano) seguindo a Convenção de commits do projeto consumidor; `/new-feature` propõe commit unificado dos artefatos de alinhamento no passo 6. Tagueamento e release, no entanto, ficam **fora desse contrato** — operador edita arquivos de versão manualmente, escreve entrada no changelog, commita e tagueia à mão a cada release.

Este repo paga o custo concretamente. `git log` mostra `chore(release): bump version to 1.10.0`, `chore(release): bump version to 1.9.0`, etc. — toda release exige edição coordenada de três arquivos (`.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, `CHANGELOG.md`) seguida de commit + tag. CHANGELOG.md em [1.10.0] inclusive carrega evidência de bug recente desse fluxo manual: dois blocos `### Added` consecutivos em vez de um (provável copy-paste imperfeito durante edição).

A pista mecânica de bump está no próprio histórico: 100% dos últimos commits seguem Conventional Commits em inglês (CLAUDE.md: "git log shows a stable pattern of Conventional Commits in English — keep it"). `feat:` → minor, `fix:` → patch, `feat!:`/`BREAKING CHANGE:` → major é inferência mecânica direta — não há ambiguidade que justifique pergunta livre.

A frase-tese da filosofia rejeita cerimônia tática. Aqui o teste passa: dor real (dois bugs visíveis no histórico recente), substituição não prevista, e a alternativa "skill que mecaniza um workflow já estabelecido" tem custo de manutenção baixo (≤200 linhas de prosa numa skill, alinhada à mecânica das outras quatro skills do plugin).

**Bifurcação arquitetural já resolvida** (registrada antes do plano por convenção; ver `docs/philosophy.md` → "Nomear bifurcações arquiteturais"):

- **(a) Caminho default-barato — skill só faz commit + tag.** Operador edita arquivos de versão e changelog manualmente antes de invocar. Custo de implementação mínimo, mas mantém a maior parte da dor original (edição coordenada de 3 arquivos error-prone).
- **(b) Caminho rico — skill mecaniza bump de version_files + entrada de changelog + commit + tag.** Maior custo de prosa na skill, mas elimina a classe inteira de bugs de release humano (versão dessincronizada entre arquivos, esquecer entrada de changelog, copy-paste imperfeito).

**Escolha: (b).** O custo dos arquivos é exatamente onde o histórico mostra erro real; pular essa parte deixaria a dor de pé. Skill detecta formato por extensão (`.json` → chave `version` top-level; `.toml` → `version = "..."`). Formatos exóticos caem em gap report — operador edita manualmente e a skill segue. Esse trade-off é o que a filosofia chama de "abstração só quando há dor real".

**Decisões já fechadas (não revisitar):**

- **Não fazer push automático.** Push é ação visível a outros (ver "Executing actions with care" do CLAUDE.md). Skill termina com a tag local; operador roda `git push --follow-tags` quando quiser.
- **Não criar GitHub Release.** Fora de escopo; `gh release create` cobre o caso.
- **Não consolidar com `/run-plan`.** Cadências diferentes — `/run-plan` commita por bloco do plano corrente; `/release` agrega N planos/blocos numa versão. Skill separada preserva a separação alinhamento → execução → release.
- **Não criar variantes stack-specific (`/release-python`, `/release-node`).** Formato de version_files é detectável por extensão; skill é genérica e segue a regra de naming do plugin (skills geram/executam stack apenas quando geram **sintaxe** ou invocam **toolchain**; `/release` faz nem um nem outro — é Git + edição estrutural de arquivos).
- **Não fazer skill alimentar dois comandos (`/bump` + `/tag`).** Cadência única — quem tagueia bump versão; separar é cerimônia.
- **Não detectar mudanças breaking de API automaticamente.** `feat!:` ou `BREAKING CHANGE:` no commit é o sinal canônico do CC. Skill respeita; operador classifica corretamente no commit.
- **Não introduzir build system / lockfile / runtime de versionamento.** CLAUDE.md veda — "Don't introduce a build system, package manager, or test runner for this repo itself."

## Resumo da mudança

Adição aditiva ao plugin:

- **Nova skill `/release`** (`skills/release/SKILL.md`). Stack-agnóstica, sem sufixo de naming. Convenção operacional (tag annotada SemVer default canonical, inferência de bump CC ≥70%, changelog como rascunho editorial, push sempre manual) **vive embutida na própria skill** — sem seção espelhada em `docs/philosophy.md`, já que a convenção é consumida por uma skill única (YAGNI; promover pra philosophy só se um segundo consumidor aparecer).
- **Dois papéis novos no path contract:** `version_files` (lista de paths a atualizar; ausência = papel desativado, sem default canonical) e `changelog` (default canonical `CHANGELOG.md`). Ambos informacionais — skill segue funcional sem nenhum dos dois (caso degenerado: só commit + tag), mas com pelo menos um dos dois entrega o valor central.
- **Atualização de `CLAUDE.md`** mencionando a nova skill e os dois papéis novos.
- **Smoke test em `docs/install.md`** cobrindo a invocação básica.

**Bump 1.10.0 → 1.11.0** — minor, aditivo. Nenhum comportamento de skill/agent/hook existente muda. Projetos consumidores que não declaram `version_files`/`changelog` no bloco de config seguem operando idêntico (skills atuais não consomem esses papéis).

**Dogfood deliberado:** o primeiro uso real da skill `/release` será para gerar a própria release **v1.11.0** que a entrega — smoke test final do plano é "skill consegue se auto-publicar". Se falhar, plano não fecha.

## Arquivos a alterar

### Bloco 1 — `skills/release/SKILL.md` (nova skill) {reviewer: code}

Criar a skill seguindo o padrão estrutural das skills existentes (`new-feature`, `new-adr`, `run-plan`, `debug`, `gen-tests-python`). Esqueleto:

- **Frontmatter:**
  - `name: release`
  - `description:` — uma frase sobre o que faz e quando usar (atualizar arquivos de versão declarados, escrever entrada no changelog, commit unificado, tag anotada `vX.Y.Z`).
  - **Sem `disable-model-invocation`** — default `false`, skill pode ser sugerida como próximo passo por outras skills/conversas. Pré-condições (working tree limpo, branch default) e confirmações enum por etapa (bump, diffs, mensagem de commit, tag) são as guard-rails reais; nenhum efeito destrutivo dispara sem confirmação explícita do operador.

- **Resumo (corpo inicial):** o que a skill faz e o que **não** faz. Frase explícita: "skill faz commit local e tag local; **não faz push** — o operador roda `git push --follow-tags` quando quiser publicar". Espelha a frase de handoff do `/new-adr`.

- **Argumentos:** opcionais — `/release` (sem argumento, infere bump do log CC) | `/release minor|patch|major` (força tipo do bump) | `/release X.Y.Z` (força versão explícita; útil para primeira release ou correção pontual). Sem argumento e sem CC extraível no log → pergunta livre ao operador.

- **Pré-condições:**
  - Resolução de papéis para `version_files` e `changelog` seguindo o protocolo padrão (`docs/philosophy.md` → "Resolução de papéis"). Ambos informacionais — skill segue mesmo com `não temos` em ambos (caso degenerado: só commit + tag).
  - **Working tree limpo (bloqueia, não cutuca).** `git status --porcelain` deve estar vazio. Release com working tree sujo gera commit que mistura release com mudança não-revisada — broken-by-construction. Mensagem de erro indica o que está sujo e direciona para `git stash` ou commit prévio.
  - **Branch é o default do projeto (cutuca, não bloqueia).** `git symbolic-ref --short HEAD` vs `main`/`master`/etc. Se diferente, enum `(a) Prosseguir nesta branch` / `(b) Cancelar e mudar para main`. Operador pode legitimamente fazer release de hotfix branch.

- **Passos:**

  1. **Detectar versão atual e calcular bump.**
     - Buscar última tag: `git describe --tags --abbrev=0 2>/dev/null` ou fallback `git tag --sort=-v:refname | head -n1`.
     - Sem tag prévia: pular para passo 1c (primeira release).
     - Listar commits desde a tag: `git log <last-tag>..HEAD --pretty=format:"%h %s"`.
     - Classificar por prefixo CC: `feat:` → minor, `fix:`/`refactor:`/`perf:` → patch, `feat!:` ou `BREAKING CHANGE:` no body → major, `docs:`/`chore:`/`style:`/`test:` sozinhos → patch (cosmético, mas valida release pequena). Critério mecânico ≥70% dos commits seguem CC para confiar; senão, cair em pergunta livre.
     - **a. Bump inferido com confiança (≥70% CC + tipo dominante claro):** enum (`AskUserQuestion`, header `Release`) com proposta como primeira opção: `(a) <bump-proposto> → vX.Y.Z` (preview da próxima versão) / `(b) Outro bump` (Other → operador especifica `minor|patch|major|X.Y.Z`). `description` de cada opção carrega o trade-off (tipo + contagem de commits que o motivou).
     - **b. Histórico CC ambíguo (<70% ou empate entre tipos):** prosa livre — mostrar resumo dos commits agrupados por prefixo, perguntar qual bump aplicar.
     - **c. Primeira release (sem tag prévia):** propor `0.1.0` como default canonical inicial via enum (`(a) 0.1.0` / `(b) 1.0.0` / Other → versão explícita).
     - Se operador passou argumento (`/release minor` ou `/release 2.0.0`), pular pergunta — argumento vence.

  2. **Atualizar `version_files` (se papel resolvido e não-vazio).**
     - Para cada path declarado, ler o arquivo e detectar formato por extensão:
       - **`.json`:** atualizar chave `version` no top-level (regex conservadora; preserva indentação).
       - **`.toml`:** atualizar primeira ocorrência de `version = "..."` no top-level. Se não encontrada, tentar dentro de `[project]` ou `[tool.poetry]` ou `[package]`. Se ainda não encontrada, gap report.
       - **Outros formatos:** gap report — skill não mexe em formato desconhecido. Reportar ao operador qual arquivo, pedir edição manual, aguardar confirmação `(a) Editado, prosseguir` / `(b) Cancelar release`. Path canonical de extensibilidade futura.
     - Mostrar diff de cada arquivo antes de aplicar via enum (`AskUserQuestion`, header `Diff`) — `(a) Aplicar` / `(b) Cancelar`. Aplicar em batch após confirmação única se diffs forem mecânicos (todos só trocaram `1.10.0` → `1.11.0`).
     - Papel resolvido para `não temos` ou lista vazia → pular passo silenciosamente.

  3. **Compor entrada de changelog (se papel resolvido).**
     - Detectar formato existente lendo as primeiras ~30 linhas. Heurística: se há linha tipo `## [X.Y.Z] - YYYY-MM-DD` no topo do conteúdo, formato é Keep-a-Changelog (default canonical). Outros formatos (changelog livre, Towncrier) → gap report (skill não tenta adivinhar; operador edita manualmente, depois confirma).
     - Compor rascunho da entrada agrupando commits por prefixo CC:
       - `feat:` → `### Added`
       - `fix:` → `### Fixed`
       - `refactor:`/`perf:` → `### Changed`
       - `docs:`/`chore:`/`style:`/`test:` → `### Notes` (compactado)
     - Cada bullet referencia o subject do commit (sem hash; muito barulho num changelog editorial).
     - Mostrar rascunho ao operador em prosa — explicitar que é rascunho e que **prosa editorial** (a "história" do que mudou, não lista crua de commits) é responsabilidade do operador. Pergunta em prosa livre: "Eis o rascunho da entrada [X.Y.Z]. Quer editar antes de aplicar, ou aplicar como está?". Espelha o passo 5 do `/new-feature` (revisão antes do commit).
     - Operador edita em prosa livre (formato: cita seção/bullet e nova redação) ou aprova como está. Skill aplica edits, insere bloco no topo do changelog (após cabeçalho `# Changelog`/intro, antes do bloco `## [versão-anterior]`).

  4. **Commit unificado.**
     - Stagear apenas os arquivos efetivamente modificados nos passos 2 e 3 (não `git add -A` — risco de capturar arquivos não-relacionados).
     - Mensagem default: `chore(release): bump version to X.Y.Z` — extraída do histórico desta repo, formato canonical para release commits em projetos CC. Se a Convenção de commits do projeto consumidor diverge (gitmoji, prefixos custom), espelhar (`docs/philosophy.md` → "Convenção de commits"; mesma mecânica de detecção das outras skills).
     - Confirmar via enum (`AskUserQuestion`, header `Commit`) — `(a) Confirmar` / `(b) Editar mensagem` (Other → mensagem alternativa).

  5. **Tag anotada.**
     - **Formato da tag detectado em três níveis** (espelho da Convenção de commits, mecânica embutida na skill):
       1. Política explícita declarada no projeto (CONTRIBUTING.md, hook, bloco no CLAUDE.md).
       2. Padrão observado nas últimas tags via `git tag --list --sort=-v:refname | head -n 10` — ≥70% seguindo um formato extraível (com `v`, sem `v`, prefixo custom como `release-`, etc.) vence.
       3. Default canonical SemVer 2.0: `vX.Y.Z`.
     - `git tag -a <tag> -m "Release <tag>"` por default. Operador pode customizar a mensagem da tag em prosa livre se quiser (raro — tag annotation costuma ser mecânica).
     - Confirmar via enum (`AskUserQuestion`, header `Tag`) — `(a) Criar tag` / `(b) Cancelar e desfazer commit`. Cancelar = `git reset --hard HEAD~1` (destrutivo, mas é o que o operador pediu; ainda dentro da blast radius local da skill).
     - Verificar que `vX.Y.Z` não colide com tag existente. Se colidir, gap report — skill **não** sobrescreve tag (tag mover é destrutivo de história e fora de blast radius local).

  6. **Reportar e devolver controle.**
     - Reportar ao operador: caminhos dos arquivos atualizados, hash do commit, nome da tag.
     - Frase final fixa: "Release pronta localmente. Para publicar: `git push --follow-tags` (ou `git push && git push origin vX.Y.Z`). GitHub Release (se aplicável): `gh release create vX.Y.Z`." — explicita os dois passos manuais que ficam de fora.

- **`## O que NÃO fazer`** (ao final, padrão das outras skills):
  - Não fazer push — release é local; publicação é decisão explícita do operador.
  - Não criar GitHub Release — `gh release` cobre o caso, fora do escopo da skill.
  - Não tocar arquivos de versão fora dos paths declarados em `version_files` — escopo é o que o papel define.
  - Não inferir bump de log que não segue Conventional Commits sem perguntar — falsa-confiança gera versionamento errado.
  - Não fazer release com working tree sujo — bloqueio explícito; operador stash/commit primeiro.
  - Não sobrescrever tag existente — colisão é gap report, não merge.
  - Não amend commit de release de versão já publicada — `--amend` reescreve história já compartilhada (só seguro pré-push, e mesmo assim apenas dentro da janela de fixup; release é cadência diferente).
  - Não pular o passo de changelog quando o papel está resolvido — drift entre código e changelog é exatamente a dor que motivou a skill.

## Verificação manual

Smoke test em consumer project — necessário porque a mudança é **comportamento de skill nova** e o repo do plugin não exercita skill em runtime. Aplicar a heurística B+C de 1.8 ao próprio plano: feature toca **comportamento de agente LLM** (skill é prosa interpretada) E **integração externa frágil** (chamadas a `git`), portanto enumerar cenários concretos e levantar a forma do dado real.

**Forma do dado real (pré-requisito, levantado antes da execução):**

- Conventional Commits subjects observados no histórico desta repo (últimos 50 commits): `chore(release): bump version to 1.10.0`, `feat(workflow): add enum-vs-prose convention via AskUserQuestion`, `fix(run-plan): update cross-reference to /new-feature commit step`, `chore(release): bump version to 1.9.0`, `feat(new-feature): add backlog review step before commit proposal`. Skill deve extrair o tipo (`chore`/`feat`/`fix`) corretamente — note o **escopo entre parênteses** (`(release)`, `(workflow)`, `(run-plan)`, `(new-feature)`) que faz parte do CC mas precisa ser ignorado para classificação de bump.
- Tag format observado: `1.10.0`, `1.9.0`, `1.8.0` (sem prefixo `v`). **Atenção:** este repo viola a convenção canonical SemVer (`vX.Y.Z`) — tags são apenas `X.Y.Z`. Skill deve detectar via protocolo de três níveis embutido (passo 5 da skill, camada 2: padrão observado ≥70%) e usar `1.11.0` (sem `v`) na nova tag, não `v1.11.0`. Cenário 7 abaixo cobre esse caso — bug latente se a skill hardcodar `v` no formato canonical sem detectar o padrão observado.
- Versão atual em `plugin.json`/`marketplace.json`: chave `"version": "1.10.0"` no top-level do JSON; sem aninhamento. CHANGELOG.md formato Keep-a-Changelog: bloco `## [1.10.0] - 2026-05-04`.

**Pré-requisito:** instalar o plugin localmente em consumer project com tag prévia + version_files declarado, via `/plugin install /storage/3. Resources/Projects/h3/pragmatic-dev-toolkit --scope project`. Confirmar carregamento via `/plugin` (skill `/release` aparece com versão 1.11.0).

**Cenário 1 (caminho frequente — bump minor inferido com confiança):**

- Consumer project com 5 commits desde última tag, sendo 3 `feat:` e 2 `chore:`. `version_files: ["package.json"]` declarado, `changelog: CHANGELOG.md` default.
- Invocar `/release` (sem argumento).
- **Esperado:** skill detecta ≥70% CC, classifica predominância como minor (3 feat domina), propõe via enum `(a) minor → 1.4.0` / `(b) Outro bump`. Operador escolhe (a). Skill atualiza `package.json` (`1.3.0` → `1.4.0`), gera rascunho de changelog com `### Added` para os 3 feat e `### Notes` para os 2 chore, mostra ao operador, operador edita prosa livremente, aplica. Skill commita `chore(release): bump version to 1.4.0`, tagueia `v1.4.0`. Mensagem final orienta `git push --follow-tags`.

**Cenário 2 (histórico sem CC — pergunta livre):**

- Consumer project com 8 commits desde última tag, todos com mensagens livres ("update stuff", "fix the thing", "more changes"). `version_files: ["pyproject.toml"]`.
- Invocar `/release`.
- **Esperado:** skill calcula <70% CC, **não** propõe bump via enum — cai em prosa livre, mostra resumo dos commits (sem agrupamento por tipo, pois não há classificação válida), pergunta ao operador qual bump aplicar. Operador responde "patch". Skill prossegue idêntico ao cenário 1 daqui pra frente.

**Cenário 3 (working tree sujo — bloqueio):**

- Consumer project com `package.json` modificado uncommitted e arquivo novo untracked.
- Invocar `/release patch`.
- **Esperado:** pré-condição falha. Skill reporta o que está sujo e direciona operador a stash ou commit prévio. Não avança.

**Cenário 4 (version_files null — só commit + tag):**

- Consumer project com `paths.version_files: null` no bloco de config (papel desativado). `changelog: CHANGELOG.md`.
- Invocar `/release minor`.
- **Esperado:** skill pula passo 2 silenciosamente (sem prompt de gap), gera changelog normalmente (passo 3), commita só com o changelog atualizado, tagueia. Sem update de arquivo de versão. Aceitável — operador escolheu não rastrear versão em arquivo.

**Cenário 5 (ambos null — caso degenerado, só commit + tag vazio):**

- Consumer project com `paths.version_files: null` E `paths.changelog: null`. Tag prévia existe.
- Invocar `/release patch`.
- **Esperado:** skill pula passos 2 e 3. Mas — não há nada para commitar! Skill detecta: enum (`(a) Tagar HEAD atual sem commit novo` / `(b) Cancelar release`). Operador escolhe (a). Skill cria tag anotada apontando para HEAD existente. Caso de borda raro mas válido — operador pode usar como tag-only workflow.

**Cenário 6 (primeira release — sem tag prévia):**

- Consumer project recém-iniciado, com 12 commits e nenhuma tag. `version_files: ["package.json"]` (versão atual `0.0.0`), `changelog: CHANGELOG.md` (vazio ou cabeçalho apenas).
- Invocar `/release`.
- **Esperado:** skill detecta ausência de tag, propõe via enum `(a) 0.1.0` / `(b) 1.0.0` / Other. Operador escolhe `0.1.0`. Skill considera todos os commits do log como "primeira entrada", gera changelog inicial, atualiza `package.json` (`0.0.0` → `0.1.0`), commita, tagueia.

**Cenário 7 (tag format não-canonical — dogfood neste repo):**

- Repo do plugin (este). Tag prévia `1.10.0` (sem `v`). `version_files: [".claude-plugin/plugin.json", ".claude-plugin/marketplace.json"]`. CHANGELOG Keep-a-Changelog.
- Invocar `/release minor` no working tree do próprio plugin após blocos 1-4 deste plano serem aplicados e commitados.
- **Esperado:** skill detecta padrão de tag observado (≥70% das tags são `X.Y.Z` sem `v`), usa `1.11.0` na nova tag — **não** `v1.11.0`. Atualiza ambos os JSONs (versão `1.10.0` → `1.11.0` espelhada). Gera rascunho de changelog do v1.11.0 cobrindo skill nova + papéis novos + nova seção em philosophy + smoke test em install + dois bullets em CLAUDE.md. Operador edita prosa para alinhar ao tom dos blocos anteriores (Notes mencionando o ciclo de disciplina fechado, dogfood). Commita `chore(release): bump version to 1.11.0`, tagueia `1.11.0`. **Este cenário é o gate final do plano** — `/run-plan` declara done apenas após este cenário passar e a tag estar criada localmente.

**Cenário 8 (operador cancela na confirmação de tag — undo do commit):**

- Consumer project, fluxo normal até o passo 4 (commit feito). No passo 5, operador escolhe `(b) Cancelar e desfazer commit`.
- **Esperado:** skill executa `git reset --hard HEAD~1`, removendo o commit de release. Working tree volta ao estado pré-release. Sem tag criada. Reportar ao operador o que foi desfeito.

**Cenário 9 (colisão de tag):**

- Consumer project, mas tag `v1.4.0` já existe (operador rodou release antes e está tentando refazer).
- Invocar `/release minor` que resultaria em `v1.4.0`.
- **Esperado:** skill detecta colisão antes do passo 5, gap report — **não** sobrescreve tag (destrutivo de história). Reporta ao operador, sugere `git tag -d v1.4.0` se intencional ou `/release patch` para `v1.4.1`. Skill aborta sem commitar.

**Cenário 10 (tag de hotfix branch — cutuca, não bloqueia):**

- Consumer project em branch `hotfix/bug-123`, não em `main`. Working tree limpo, tag prévia `v1.4.0`.
- Invocar `/release patch`.
- **Esperado:** pré-condição "branch é o default" falha. Skill cutuca via enum `(a) Prosseguir nesta branch` / `(b) Cancelar e mudar para main`. Operador escolhe (a). Skill prossegue idêntico ao cenário 1.

**Critério de aprovação:** todos os 10 cenários exibem o comportamento esperado. **Cenário 7 é o crítico** — é o smoke test em produção e simultaneamente o ato de release deste plano. Cenários 3, 5, 9 confirmam guards (não fazer release com working tree sujo, não fazer release vazia inesperada, não sobrescrever tag). Cenários 4, 5, 8, 10 confirmam que a skill é maleável aos casos de borda sem rigidez.

## Notas operacionais

- **Reviewer escolhido em todos os blocos: `code` (default).** Sem invariantes de domínio (`qa`) — o repo do plugin não tem `docs/domain.md`, e a skill `/release` não testa comportamento de domínio do consumidor. Sem superfície de segurança nova (`security`) — skill **não** lida com segredos, autenticação, validação de input externo nem privilégios; opera só sobre o working tree local e Git. Push é o único vetor potencial de exposure, e é deliberadamente excluído ("Não fazer push" em `## O que NÃO fazer` da skill). Rubrica YAGNI do `code-reviewer` é a relevante para esta mudança editorial-mecânica.
- **Plano é meta** — refina o próprio plugin. Não há `test_command` automatizado (CLAUDE.md confirma). Validação manual em consumer project + dogfood no cenário 7 substituem o gate automático.
- **Dogfood obrigatório.** O bloco 5 (bump CHANGELOG/manifests) **não** é editado manualmente durante a execução. `/run-plan` aplica blocos 1-4 + commits unificado por bloco (`code-reviewer` em cada um); ao final, em vez de declarar done com bloco 5 pendente, a verificação manual cenário 7 invoca `/release minor` no working tree, que gera o bloco 5 mecanicamente. Se a skill falhar nesse cenário, o plano é incompleto e `/run-plan` não declara done.
- **`.worktreeinclude` não tocado** — release não tem credencial gitignored. Worktree do plano replica apenas arquivos versionados, comportamento padrão.
- **Itens fora-de-escopo emergidos na conversa:** nenhum. Operador alinhou apenas ao escopo da skill `/release` + papéis novos + convenção. Possibilidades descartadas (push automático, GitHub Release, separar `/bump` e `/tag`, variantes stack-specific, detecção breaking automática) registradas em "Decisões já fechadas" para evitar revisitar.
- **Bump 1.11.0 fecha simetria** com a disciplina de commits já existente. Eixo "release" passa a ter mecânica explícita; alinhamento → execução → release vira contrato de três estágios espelhando alinhamento → execução → revisão que `/run-plan` já entrega por bloco. CHANGELOG `### Notes` deve mencionar essa coerência.
