# v1.8 — fortalecer gate de `## Verificação manual`: cenários enumerados e credenciais replicadas

## Contexto

Sessão real de uso entregou feature em projeto Python que envolveu (a) matching de strings com dado real do cliente e (b) ajuste de prompt de agente LLM. Reviewers automáticos passaram. **O gate de `## Verificação manual` foi o que pegou 3 bugs reais** — mas os 3 expuseram pontos cegos no fluxo atual:

1. **Worktree subiu sem `.env`.** Baseline (unit tests) verde, mas a verificação manual exigia `make telegram` que precisa de credenciais. `/run-plan` passo 1.2 só propõe `.worktreeinclude` quando o arquivo **não existe** — não cruza com o fato do plano ter `## Verificação manual` que sobe serviço real.

2. **Matching falhou em formato realista de produção.** Testes cobriram tokens isolados (`"inter"`, `"pix"`); cliente real digitou `"pix inter"` contra cadastro `"PIX - INTER"`. Ninguém perguntou *"como o dado é nomeado em produção?"* — plano e `qa-reviewer` aceitaram a cobertura sintética.

3. **Agente vazou ids internos no texto da resposta ao usuário.** Regra estava diluída no corpo do prompt; só virou explícita após validação manual mostrar o vazamento. `## Verificação manual` dizia "validar via interface real" sem enumerar cenários — operador improvisou e descobriu o bug por sorte.

Os três apontam para o **mesmo lugar**: enumeração fraca de cenários em `## Verificação manual` quando feature toca surface não-determinística (parsing, matching, agente LLM). Bug 1 é instância específica (`.worktreeinclude` reativo demais).

**Decisões já fechadas (não revisitar):**
- Não criar agente `prompt-reviewer` — uma sessão não justifica papel novo (YAGNI).
- Não criar seção `## Cenários do agente` no template de plano — criaria precedente para seção por tipo de feature; `## Verificação manual` é o guarda-chuva certo.
- Não mudar mandato do `qa-reviewer` — reviewer age sobre diff, não conhece "dado real". Defesa fica upstream em `/new-feature`.

## Resumo da mudança

Três refinamentos aditivos, todos dentro de skills existentes (sem agente novo, sem seção nova de plano):

- **A (`/run-plan`):** passo 1.2 ganha gatilho mais forte. Quando plano corrente tem `## Verificação manual` E há credenciais gitignored típicas (`.env`, `*.local.yaml`) **não replicadas na worktree**, propor `.worktreeinclude` (criar ou aumentar) antes do baseline. Fio condutor unificador: *credencial necessária para validação manual não está replicada*. Os 3 casos do mundo real (`.worktreeinclude` ausente / `.worktreeinclude` existe sem a credencial / operador disse "não preciso" antes mas plano corrente exige validação) são instâncias do mesmo cruzamento. Justificar ao operador *por que agora* (validação manual vai precisar do serviço real).

- **B+C (`/new-feature`):** gap existente *"Validação manual necessária?"* no checklist do passo 2 ganha **sub-bullet condicional** (não dois itens novos no top-level — refinamento de heurística > expansão da lista, alinhado a "YAGNI por padrão" e ao precedente de `/run-plan` 4.3 que aninhou condições no mesmo gap). Quando a resposta for "sim" E a feature tocar parsing, matching de strings, ou comportamento de agente LLM, exigir antes do plano: (a) 1-2 exemplos concretos da forma do dado real em produção (separadores, prefixos, capitalização inconsistente); (b) `## Verificação manual` enumerar cenários concretos que exercitem essas formas, não direção genérica tipo "validar via interface real".

- **Release:** bump 1.7.0 → 1.8.0 (mudança aditiva, complementa o eixo "gate de validação manual" iniciado em 1.5/1.6/1.7); CHANGELOG entry no padrão estabelecido (`### Added` + `### Notes`).

Não há mudança no `philosophy.md` — a heurística vive nas skills, e `philosophy.md` não tem seção dedicada a "Validação manual" como conceito (só referenciada de passagem). Não há mudança em `CLAUDE.md` deste repo — descreve o workflow das skills em alto nível e não cita os pontos refinados.

## Verificação manual

Smoke test em consumer project — necessário porque a mudança afeta o comportamento das skills no runtime e o repo do plugin não tem como exercitar isso. Aplicar a heurística B+C ao próprio plano: feature toca **comportamento de agente LLM** (skills são prosa interpretada), portanto enumerar cenários concretos. Cenários abaixo, em ordem:

**Pré-requisito:** instalar o plugin localmente em um consumer project com `/plugin install /storage/3. Resources/Projects/h3/pragmatic-dev-toolkit --scope project`. Confirmar carregamento via `/plugin` (ambas as skills aparecem com versão 1.8.0).

**Cenário 1 (refinamento A — caso `.worktreeinclude` ausente):**
- Consumer project sem `.worktreeinclude` na raiz; `.gitignore` lista `.env`; existir um plano em `docs/plans/teste-manual.md` com seção `## Verificação manual`.
- Invocar `/run-plan teste-manual`.
- **Esperado:** antes do baseline, skill detecta `## Verificação manual` + `.env` gitignored não coberto e cutuca: *"Plano tem `## Verificação manual` e `.env` não está replicada na worktree. Validação manual provavelmente vai precisar do serviço real. Replicar agora?"*

**Cenário 2 (refinamento A — caso `.worktreeinclude` incompleto):**
- Mesmo consumer, `.worktreeinclude` existindo com apenas `node_modules` listado; `.env` gitignored.
- Invocar `/run-plan teste-manual` com plano que tem `## Verificação manual`.
- **Esperado:** skill detecta `.env` não coberto (apesar de `.worktreeinclude` existir) e cutuca igual ao Cenário 1, propondo adicionar `.env` ao `.worktreeinclude` existente.

**Cenário 3 (refinamento A — caso "não preciso" anterior + plano com `## Verificação manual`):**
- Mesmo consumer, `.worktreeinclude` existindo vazio (operador declarou "não preciso" em invocação anterior); `.env` gitignored.
- Invocar `/run-plan teste-manual` com plano que tem `## Verificação manual`.
- **Esperado:** skill **re-pergunta** sobre `.env` (apesar do `.worktreeinclude` vazio existir), justificando que o estado prévio é silenciado quando `## Verificação manual` está presente.

**Cenário 4 (refinamento A — caso negativo, sem `## Verificação manual`):**
- Mesmo consumer, `.worktreeinclude` ausente; `.env` gitignored; plano em `docs/plans/refactor-puro.md` **sem** `## Verificação manual`.
- Invocar `/run-plan refactor-puro`.
- **Esperado:** skill **não dispara** a cutucada cruzada (sem `## Verificação manual`, gatilho não bate). Comportamento legado preservado: pode ainda propor `.worktreeinclude` pela cláusula original ("ausente, primeira vez"), mas com mensagem genérica, não justificando "validação manual vai precisar".

**Cenário 5 (refinamento B+C — caso parsing/matching dispara):**
- Em qualquer consumer, invocar `/new-feature normalizar formatos de data digitados pelo usuário`.
- **Esperado:** durante o passo 2, skill identifica gatilho de parsing/matching e pergunta: *"Forma do dado real? 1-2 exemplos concretos de formato que usuários digitam em produção (separadores, capitalização)."* Após resposta, plano resultante tem `## Verificação manual` com cenários enumerados (ex.: "validar input `01/05/2026`", "validar input `2026-05-01`", "validar input `1 de maio`") e não direção genérica.

**Cenário 6 (refinamento B+C — caso agente LLM dispara):**
- Invocar `/new-feature ajustar prompt do agente para nunca incluir id interno na resposta ao usuário`.
- **Esperado:** skill identifica gatilho de "comportamento de agente LLM", pergunta sobre forma do dado real (ids internos típicos que aparecem no prompt) e plano tem `## Verificação manual` enumerando cenários (ex.: "perguntar sobre transação X, verificar resposta não contém `txn_id`", "pedir resumo, verificar resposta não contém `user_id`").

**Cenário 7 (refinamento B+C — caso negativo, sem trigger):**
- Invocar `/new-feature ajustar cor do botão de download`.
- **Esperado:** skill **não pergunta** sobre forma do dado real e plano (se houver `## Verificação manual`) pode ter direção mais simples — sub-bullet condicional não dispara para feature que não toca parsing/matching/LLM.

**Cenário 8 (refinamento B+C — caso negativo, validação manual = não):**
- Invocar `/new-feature refatorar cálculo interno de hash sem mudar contrato`.
- **Esperado:** primeira pergunta do gap "Validação manual necessária?" resolve "não" (refactor puro, sem comportamento user-facing); sub-bullet condicional **não** dispara mesmo se o operador mencionar "matching" no contexto. Refinamento B+C é refinamento do gap de validação manual, não trigger orthogonal.

**Critério de aprovação:** todos os 8 cenários exibem o comportamento esperado. Cenários 4, 7 e 8 são especialmente importantes — confirmam que o refinamento **não** introduz cerimônia onde não se aplica.
